# Parity Evidence Records

Store release-candidate parity evidence records in this directory.

Use one Markdown file per adapted family and environment class, copied from
`../PARITY_EVIDENCE_TEMPLATE.md`. Record files should be named with the family,
environment, and candidate identifier, for example:

```text
layout-kitty-0123456789abcdef.md
remote-delivery-browser-abcdef1234567890.md
```

For `Remote-delivery`, start from
`../REMOTE_DELIVERY_EVIDENCE_TEMPLATE.md` when the evidence includes a real
browser/WebSocket deployment. The focused template preserves the same family and
artifact expectations while adding security, protocol, browser, and deployment
fields required for release review.

The closeout audit requires the filename to include the normalized family name,
terminal or browser environment, and release-candidate commit recorded in the
file.

The machine-readable policy for these records is stored in
`parity_policy.json`. The quality gate checks that it contains the reviewed
families, required fields, required sections, Linux scope, and draft/final
directories. It also stores the allowed command entrypoints, artifact URL
schemes, and manual artifact hints for final evidence records. Manual artifact
hints include `terminal`, `manual`, `transcript`, `screenshot`, `recording`, and
`matrix`. `schema_version` changes only when the policy shape changes in a way
future tooling must handle explicitly. Keep that file, the scaffold, closeout
audit, and quality gate aligned when adding or changing adapted parity families.

Run the policy audit when changing families, scopes, checklist text, or scaffold
support:

```sh
julia --project=. --startup-file=no scripts/parity_policy_audit.jl
```

## Machine-readable release artifacts

This directory also stores schemas for generated CI artifacts used during widget
stabilization and release review.

| Schema | Generated artifact | Audit |
|---|---|---|
| `stable_widget_coverage.schema.json` | `ci-artifacts/stable-widget-coverage-summary.json` | `scripts/stable_widget_coverage_schema_audit.jl` |
| `stable_widget_stability.schema.json` | `ci-artifacts/stable-widget-stability.json` | `scripts/stable_widget_stability_schema_audit.jl` |
| `stable_widget_stabilization.schema.json` | `ci-artifacts/stable-widget-stabilization.json` | `scripts/stable_widget_stabilization_schema_audit.jl` |
| `stable_widget_surface_release.schema.json` | `ci-artifacts/stable-widget-surface-release.json` | `scripts/stable_widget_surface_release_schema_audit.jl` |
| `widget_promotion_requirements.schema.json` | `ci-artifacts/widget-promotion-requirements.json` | `scripts/widget_promotion_requirements_schema_audit.jl` |
| `reference_parity_matrix.schema.json` | `ci-artifacts/reference-parity-matrix.json` | `scripts/reference_parity_matrix_schema_audit.jl` |
| `reference_parity_summary.schema.json` | `ci-artifacts/reference-parity-summary.json` | `scripts/reference_parity_matrix_schema_audit.jl` |
| `reference_parity_matrix_status.schema.json` | `ci-artifacts/reference-parity-matrix-status.json` | `scripts/reference_parity_matrix_schema_audit.jl` |

The promotion requirements artifact is generated from
`api/widget_promotion_requirements.tsv` by
`scripts/render_widget_promotion_requirements.jl --format json --release-required yes`.
It lets release dashboards consume the same experimental-to-stable promotion
contract that reviewers inspect as Markdown.

The stable widget stability artifact is generated by
`scripts/render_widget_catalog.jl --stability-json`. It lets dashboards consume
the same promotion-readiness blockers that reviewers inspect in
`ci-artifacts/stable-widget-stability.md` and
`ci-artifacts/stable-widget-stability-gaps.md`.

The stable widget surface-release artifact is generated by
`scripts/render_widget_catalog.jl --surface-release-json`. It lets dashboards
consume the combined release decision that reviewers inspect in
`ci-artifacts/stable-widget-surface-release-status.txt`. The schema audit checks
that `release_ready` agrees with stable widget coverage readiness, widget
stability readiness, family closeout readiness, and git provenance.

The reference parity matrix artifact is generated from
`docs/REFERENCE_PARITY_SURVEY.md` by `scripts/render_reference_parity_matrix.jl`.
Its schema audit checks both row keys and summary arithmetic, so
`summary.total` and `summary.by_status` must agree with the generated rows.
Rows marked `adapted`, `intentional divergence`, or `not yet implemented` must
also cite a release-checklist item or issue follow-up. Release mode for the
schema audit rejects any row that is not marked `matched`; adapted rows must
first receive final parity closeout evidence and then be promoted to `matched`
before release status can be green.
`scripts/parity_closeout_audit.jl --require-complete` derives its required
families from those non-matched survey rows, maps them through
`parity_policy.json`, and requires at least
`minimum_final_records_per_family` valid final records for each currently
blocking family.
The same audit can render closeout requirement reports showing required,
observed, and missing final records per blocking family. Each row includes a
draft `scripts/new_parity_evidence.jl` scaffold command for the family:

```sh
julia --project=. --startup-file=no scripts/parity_closeout_audit.jl --report markdown
julia --project=. --startup-file=no scripts/parity_closeout_audit.jl --status
julia --project=. --startup-file=no scripts/parity_closeout_audit.jl --report tsv
julia --project=. --startup-file=no scripts/parity_closeout_audit.jl --report json
```

Pass `--family <family>` to focus any report or status output on one blocking
family. The filter accepts policy names such as `Remote-delivery` and survey
names such as `Remote delivery`, case-insensitively. Unknown family filters fail instead of
emitting an empty report.

The JSON report is described by
`docs/evidence/parity_closeout_requirements.schema.json`.

CI also writes `ci-artifacts/reference-parity-matrix-status.txt` and
`ci-artifacts/reference-parity-matrix-status.json`, compact release-readiness
artifacts derived from the same matrix. The text status can be rendered with
`--require-release-ready` to fail release scripts when any family remains
outside `matched`. Use `--release-blockers` when a reviewer needs one
newline-separated list of the remaining family closeout targets without the full
matrix columns. The status JSON schema requires
`release_ready`, `total`, `blocking`, `blocking_families`, and
`blocking_records`; the schema audit checks that those counts agree and that the
detailed blocker records identify the same families as the compact family list.
For release review, CI also writes
`ci-artifacts/reference-parity-review.md`, a focused Markdown view containing
only `family`, `status`, and `follow_up`, plus
`ci-artifacts/reference-parity-blocking.md` and
`ci-artifacts/reference-parity-blocking.json`, focused views of only non-matched
release-blocking rows rendered with `--blocking-only`, plus
`ci-artifacts/reference-parity-adapted.md`, the adapted-family implementation
and evidence backlog, plus `ci-artifacts/reference-parity-adapted.json` for
dashboards. The adapted and blocking JSON artifacts use the same schema as
`reference-parity-matrix.json`; the schema audit verifies that adapted rows
remain filtered to `adapted` and blocking rows never contain `matched` status.
The parity closeout requirements report carries each blocking survey row into
the implementation policy view with `survey_family`, `parity_status`, and
`follow_up`, then pairs that context with the exact evidence scaffold command
needed to promote the affected widget family out of experimental status.
When the full matrix still reports adapted or blocking work, the audit also
verifies that those focused backlogs are not empty. CI also writes
`ci-artifacts/reference-parity-remote-delivery.md` for the browser/protocol
parity scope and `ci-artifacts/reference-parity-remote-delivery.json` for
dashboards. The schema audit verifies that the remote-delivery JSON remains
filtered to that family. CI also writes
`ci-artifacts/reference-parity-summary.tsv` and
`ci-artifacts/reference-parity-summary.json` with status counts.

Run the closeout audit to validate every committed final record in this
directory:

```sh
julia --project=. --startup-file=no scripts/parity_closeout_audit.jl
```

Before claiming production parity for a release candidate, require at least one
valid final record for every reviewed adapted family. The exact threshold is
stored as `minimum_final_records_per_family` in `parity_policy.json`:

```sh
julia --project=. --startup-file=no scripts/parity_closeout_audit.jl --require-complete
```

`Core rendering` is not listed here because the reference survey marks it as
`matched`, not adapted. Core rendering still requires release-candidate evidence,
but that evidence is recorded through the core API tests, terminal byte tests,
Unicode/width corpus, Linux real-terminal matrix, and release checklist rather
than through an adapted-family migration record.

## Create a record

1. Create a draft record with the scaffold script or copy the template manually:

List the current release-blocking parity families first:

```sh
julia --project=. scripts/new_parity_evidence.jl --list-blocking
```

```sh
julia --project=. scripts/new_parity_evidence.jl \
  --family Layout \
  --environment kitty-truecolor \
  --candidate 0123456789abcdef
```

By default, the scaffold writes to `scratch/parity-evidence/`, which is ignored
and safe for incomplete drafts. Use `--final true` only when creating a completed
record directly under `docs/evidence/`. Final records require every identity
field plus `--behavior`, `--summary`, `--parity-notes`, and `--risks` so the
scaffold does not write TODO placeholders or empty required sections into the
quality-gated evidence directory. `--exit-status` must be a non-negative integer
process exit code. `--artifact` must use an artifact URL scheme from
`allowed_artifact_url_schemes` in `parity_policy.json`, currently `http://` or
`https://`, or a local artifact path that exists relative to the repository
root; the scaffold rejects placeholder artifact references in final mode.
`--command` must be the exact command that produced the evidence, not a placeholder. Automated commands
must reference a repository evidence entry point such as `scripts/`, `test/`,
`benchmark/`, `docs/make.jl`, `Pkg.test`, or `node --check`; manual records must
start the command field with `manual:`. The authoritative command entrypoint
list is stored in `required_command_entrypoints` in `parity_policy.json`.
Manual evidence artifact paths or URLs must include one of the
`manual_artifact_hints` values from the policy, such as `terminal`,
`transcript`, `screenshot`, `recording`, or `matrix`.
`--behavior` must include the exact family closeout scope from
`parity_policy.json`, matching the closeout audit.
`--environment` must identify the terminal, browser, or CI environment. `--date`
must use `YYYY-MM-DD HH:MM:SS UTC`. `--julia-version` must be a Julia version
such as `1.10.11` or `1.12.6`. `--kernel` must identify a Linux kernel or
distribution.

```sh
julia --project=. scripts/new_parity_evidence.jl \
  --family Layout \
  --environment kitty-truecolor \
  --candidate 0123456789abcdef \
  --final true \
  --date '2026-07-12 12:00:00 UTC' \
  --julia-version '1.12.6' \
  --kernel 'Linux 7.0.0 x86_64' \
  --capability 'UnicodeWidthPolicy, truecolor' \
  --command 'julia --project=. --startup-file=no scripts/widget_audit.jl --require-complete' \
  --exit-status 0 \
  --artifact https://github.com/acme/wicked/actions/runs/123456789 \
  --behavior 'constraint edge cases, clipping policy, resize continuity, narrow-terminal behavior were exercised' \
  --summary 'strict widget audit completed without missing layout evidence' \
  --parity-notes 'observed behavior matches Ratatui-style deterministic layout with documented Wicked clipping policy' \
  --risks 'real-terminal glyph and resize behavior still require the manual terminal matrix'
```

Replace the illustrative CI URL above with the actual artifact URL for the
release-candidate run before creating a real final record. The quality gate
rejects placeholder artifact references in committed records.

2. If copying manually, copy `../PARITY_EVIDENCE_TEMPLATE.md` into this directory
   with a descriptive lowercase filename.
3. Fill every identity field before committing the record.
4. Use one of the reviewed family names exactly: `Layout`, `Input-event`,
   `Stateful-controls`, `Data-display`, `Runtime`, `Developer-experience`,
   `Styling-theming`, or `Remote-delivery`.
5. Record the exact command, exit status, candidate commit SHA, and artifact or
   CI URL. The scaffold accepts short or full hexadecimal git commit SHAs only.
6. List the checked behaviors using the exact release-closeout scope for the
   family from `parity_policy.json`; the closeout audit rejects generic behavior
   summaries that do not include the policy scope.
7. Attach or link the transcript, snapshot approval, benchmark artifact, browser
   run, terminal run, or CI output that proves the claim.

## Reviewed adapted families

Use these family names exactly in the `Family` identity field:

| Family | Release-closeout scope |
| --- | --- |
| `Layout` | Constraint edge cases, clipping policy, resize continuity, and narrow-terminal behavior |
| `Input-event` | Routed events, async delivery, cancellation behavior, focus restoration, and terminal lifecycle recovery |
| `Stateful-controls` | Widget contract tests, state-transition tests, semantic snapshots, and stable widget candidate evidence |
| `Data-display` | Virtual list/table/tree stress cases, stale data, loading/error slots, and screen-reader semantic state |
| `Runtime` | Queue replacement, task cancellation races, redraw determinism, resource cleanup, and subscription shutdown |
| `Developer-experience` | API contract tests, Pilot/semantic query evidence, migration notes, examples, and documentation build output |
| `Styling-theming` | Selector specificity, cascade order, role downgrade behavior, diagnostics, and monochrome fallback |
| `Remote-delivery` | Browser deployment, WebSocket hardening, protocol versioning, security policy, and real-client compatibility |

The scaffold and quality gate both reject family names outside this reviewed
set. Add new adapted families to the reference survey, release checklist, and
quality gate in the same change. Also update `parity_policy.json` so tooling,
documentation, and validation use the same family and scope list.

The quality gate also checks that `scripts/new_parity_evidence.jl` remains
available, continues to expose the required draft and final-record fields, and
keeps scaffolded drafts ignored under `scratch/parity-evidence/`.
`scripts/parity_policy_audit.jl` checks that the policy, scaffold, README, and
release checklist use the same family names and closeout scopes.
`scripts/parity_closeout_audit.jl` validates committed final records and, in
`--require-complete` mode, rejects release candidates that do not have final
records for every reviewed adapted family. It also rejects final records whose
`Behaviors checked` section does not include the exact family closeout scope
from `parity_policy.json`.

Scaffolded records are drafts unless created with `--final true`. Replace every
`TODO` value and fill every blank identity field before expecting the quality
gate to accept the record. The quality gate rejects committed evidence records
containing `TODO` anywhere in the file.

The quality gate validates every record in this directory except this README. It
checks that required template sections are present, required identity fields are
filled, command and artifact references are not placeholders, exit status is a
non-negative integer, release-candidate commit is a short or full hexadecimal
git SHA, artifact references are HTTP(S) URLs or existing local artifact paths,
environment and release identity fields are not placeholders, the date uses
`YYYY-MM-DD HH:MM:SS UTC`, kernel identity is Linux, the family name is one of
the reviewed adapted families, Julia version uses a semver-like format,
`Behaviors checked` includes the policy closeout scope for the family, and all
narrative sections contain concrete non-placeholder content rather than empty
bullets or punctuation. Reference-library parity notes must mention Ratatui,
Textual, TamboUI, Lanterna, or an explicit intentional divergence.

No file in this directory is release evidence until it references an immutable
release-candidate commit and the associated command output, CI URL, terminal
transcript, browser artifact, or benchmark artifact.

Do not use this directory for notes, drafts, local-only experiment logs, or
unreviewed terminal screenshots. Keep those outside the repository until they are
ready to become release-candidate evidence.
